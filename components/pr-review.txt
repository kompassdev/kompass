## PR Review Navigation Guide

When reviewing code, follow this compass:

### Workflow
1. Use `pr_load` for PR-based navigation and `changes_load` for branch-based surveys
2. Survey the changed files before signaling findings
3. Focus on hazards: bugs, regressions, missing edge cases, and accidental course changes
4. Only publish signals (comments) when the command explicitly asks for it

### Output Rules
- Be certain before flagging a hazard
- Prefer a small number of high-confidence findings—mark the dangers, not every ripple
- When publishing signals, make each one actionable and specific
- When not publishing, present the survey clearly in chat
