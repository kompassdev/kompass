## PR Review Navigation Guide

When reviewing code, optimize for high-signal findings per token.

### Workflow
1. Start with a thin loader: use `changes_load` for git-based change inspection and `pr_load` only for PR metadata, reviews, comments, and reviewer identity
2. Check `AGENTS.md` guidance before reviewing behavior or conventions; when changed files live under nested directories, use the most specific nested `AGENTS.md` that governs that path
3. Scan the summary first so you know the scope, file states, commit history, and where the risk clusters are
4. Use `changes_load` to get the changed-file set and structured per-file diffs; do not rely on `pr_load` for file changes
5. Do not expand every file by default; inspect diffs and full files selectively, prioritizing risky files and suspicious paths first
6. Read the full current file for added or modified files before making a claim; for deleted files, inspect the previous file contents from git; for renamed files, validate the move and all affected references
7. In CI or shallow clones, prefer `changes_load` with explicit base and head refs so it can fetch the missing git history it needs
8. Follow related code only when it is necessary to confirm behavior, prove a regression, or verify compliance with project guidance
9. Only publish signals when the command explicitly asks for it

### Efficient Expansion
- Use the loader output to decide which files deserve deeper inspection; do not blindly read every changed file in large PRs
- Prefer `changes_load` structured per-file diffs first, then targeted git diff commands for deeper inspection when needed
- Read only the files that are likely to contain the root cause, not every downstream caller
- Skip generated, lockfile, or bulk-format churn unless it hides a meaningful source change nearby
- If the first-pass summary already proves there is no issue, move on instead of spending more tokens

### What To Look For
- behavior that changed accidentally, not just code that looks unusual
- incorrect assumptions about null, empty, default, retry, timeout, ordering, and permission paths
- caller and contract breakage: API shapes, return values, schema changes, renamed exports, moved files, and config semantics
- partial updates: code path changed in one place but not in another dependent path
- deleted or renamed files that leave stale references, docs, routes, imports, migrations, or tests behind
- risky generated or lockfile churn that may hide a real source change nearby

### Finding Threshold
- Report a finding only when you can name the likely failure mode.
- Report a convention or workflow violation only when it conflicts with the applicable `AGENTS.md` guidance.
- Do not report style, naming, or minor cleanup unless it masks a real defect.
- If something looks suspicious but unproven, call it out as a question only when the workflow asks for a chat summary; do not publish speculative inline comments.
- Be concise; if a point is not important enough to state in a few direct sentences, it probably should not be a finding.

### Importance Guide
- `critical`: data loss, security exposure, broken deploy or migration, severe customer impact
- `high`: likely bug or regression in normal usage, broken contract, missing validation on an important path
- `medium`: edge-case bug, incomplete handling, or maintainability issue likely to cause follow-up breakage
- `low`: useful but non-blocking improvement; avoid publishing these unless they are still worth interrupting the author for

### Output Rules
- Be certain before flagging a hazard
- Prefer a small number of high-confidence findings—mark the dangers, not every ripple
- Follow `AGENTS.md` and nested `AGENTS.md` guidance strictly when deciding whether something is compliant
- When publishing signals, make each one actionable, specific, and importance-graded
- For PR reviews, publish comments by default; only approve when the final grade is `★★★★★`
- Do not duplicate prior feedback; if the same issue was already flagged, skip it
- Do not repeat the same grade; if the grade is unchanged, add only genuinely new comments or skip publishing entirely
- When reporting a grade, format it as stars like `★★☆☆☆`
- When not publishing, present the survey clearly in chat with a star grade, a short overall verdict, and a concise finding list ordered by impact

### Publishing Reviews
- Use `pr_load.viewer.login` plus the returned review and comment history to detect your own prior feedback before posting anything
- Treat a comment as duplicate when you already flagged the same issue on the same file location, even if the wording differs slightly
- Compare the new grade against your latest prior grade from the same reviewer identity before posting a review summary
- If the grade is unchanged and there are no new comments, skip publishing entirely
- If the grade is unchanged and there are new comments, submit only inline comments with `event: COMMENT` and no review body
- If the grade changed, submit a review body with the star grade; use `APPROVE` only for `★★★★★`, otherwise use `COMMENT`
- Publish through `gh api --method POST repos/{repo}/pulls/{pr}/reviews --input <json-file>`
