## Purpose

Use `/ticket/plan` to shape a grounded implementation plan before any code changes happen.

## Usage

```text
/ticket/plan <ticket-reference-or-request>
```

## Typical flow

- load ticket comments when planning from a ticket, otherwise use the request and conversation
- derive the planning objective, constraints, technical direction, and open questions
- do light repo reconnaissance for technical work
- return a title, short plan description, implementation checklist, and validation checklist without syncing ticket state

## Common tools

- `ticket_load`
- `read`
- `grep`
